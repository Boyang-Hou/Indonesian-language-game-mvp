# MVP Demo 需求文档 · 印尼语沉浸式学习平台

> 版本 0.0.2 · 验证阶段 · 纯前端 Demo
>
> 目标：用最小成本验证核心假设，不是做产品。

---

## 一、我们在验证什么

### 核心假设

> 用视觉小说式的沉浸场景对话学印尼语，比现有资源更让初学者愿意继续。

### 验证标准

- 找 5-10 个测试用户（外语老师 + 语言学习爱好者 + 身边朋友）
- 观察指标：完成率（多少人走完第一章）、停留时长、哪个场景流失最多
- 主观反馈：用完之后会不会想要"下一章"
- **8 周观察期，止损线：完成率 < 15%、零自传播 → 暂停项目**

---

## 二、Demo 范围：只做这些，别的都不做

### 必须有

- [ ] 5 个场景，每场景 5-8 轮对话，数据 hardcode 在前端 JSON 文件中
- [ ] 印尼语对话 + 中文对照逐轮展示（点击/按钮推进下一轮）
- [ ] 生词高亮：`vocab_hints` 中标注的词在印尼语文本中以特殊样式显示
- [ ] 生词弹窗：点击高亮词弹出释义（词义 + 词根拆解 + 文化钩子一句话）
- [ ] 选项分支：每个场景最后一轮对话后展示 2-3 个选项，点击跳转到下一场景
- [ ] 第一章结尾（场景 5）展示悬念钩子，告诉用户"第二章即将到来"
- [ ] 部署到 Vercel，生成一个可分享的链接

### 现在不做，验证通过后可能做

- ⏳ 后端 / API / 数据库（验证通过后加，支撑用户系统和进度保存）
- ⏳ 用户注册 / 登录（需要后端后再接入 Supabase Auth）
- ⏳ 进度保存 / 多设备同步（依赖后端，demo 阶段刷新从头开始）
- ⏳ 任意词查询（demo 仅高亮词可点击，后续可扩展全词库）
- ⏳ 角色立绘 / 精美背景图（需要美术资源，v0.2 引入）
- ⏳ 音频 / TTS 发音（需评估 TTS API，v0.3+）
- ⏳ 词汇本 / 复习功能（依赖用户系统）
- ⏳ 单元测试 / Schema 校验器（产品化阶段补上）

### Demo 阶段明确不做（即使后续也不优先）

- ❌ XP 等级系统（增加复杂度，不影响核心学习体验验证）
- ❌ ToB 商务印尼语内容线（方向未验证前不分散精力）
- ❌ AI 对话练习模块（独立产品方向，不属于当前假设）

## 三、技术方案：越简单越好

| 层 | 方案 | 说明 |
|----|------|------|
| 框架 | React + Tailwind CSS | Vibe Coding 友好，组件化 |
| 数据 | 本地 JSON 文件 | 直接 import，不调 API |
| 部署 | Vercel | 免费，推送即部署 |
| 美术 | CSS 占位 | 渐变背景 + 文字标签，不需要图片资源 |


## 四、场景数据结构

### JSON 格式定义

```json
{
  "scene_id": "jakarta_market_02",
  "chapter": 1,
  "background_style": "linear-gradient(135deg, #F5E6CA 0%, #A8D5A2 100%)",
  "dialogues": [
    {
      "character": "Pedagang (摊贩)",
      "indo": "Selamat pagi! Mau beli apa hari ini?",
      "zh": "早上好！今天想买什么？",
      "vocab_hints": [
        {
          "word": "selamat pagi",
          "meaning": "早上好",
          "root": "selamat=平安/祝福 + pagi=早晨",
          "culture_hook": "印尼人见面必说 selamat，这个词本意是'平安'，问候语都在祝你平安"
        },
        {
          "word": "mau",
          "meaning": "想要 / 要",
          "root": "基础词，无词根拆解",
          "culture_hook": "印尼语中最高频的意愿表达词，点餐、购物、日常对话全靠它"
        }
      ]
    },
    {
      "character": "你",
      "indo": "Saya mau beli sayur. Berapa harga kangkung?",
      "zh": "我想买蔬菜。空心菜多少钱？",
      "vocab_hints": [
        {
          "word": "berapa",
          "meaning": "多少",
          "root": "基础疑问词",
          "culture_hook": "砍价必备词，在印尼菜市场你会用到无数次"
        }
      ]
    }
  ],
  "choices": [
    {
      "label": "Terlalu mahal! Bisa kurang?（太贵了！能便宜点吗？）",
      "next": "jakarta_market_03a"
    },
    {
      "label": "Oke, saya beli.（好，我买了。）",
      "next": "jakarta_market_03b"
    }
  ]
}
```

**与旧 PRD 的关键区别：**

- `dialogues` 是**数组**，不是单个对象，支持多轮对话
- 没有 `xp` 字段（不做经验系统）
- 没有 `category` 字段（不考虑 ToB 预留）
- `background_style` 直接是 CSS 值，不是图片引用
- 选项里去掉了"求助"选项，保持简洁

### 结尾场景数据结构（场景 5 特殊处理）

场景 5 没有选项分支，`choices` 为空数组，新增 `ending` 字段承载结尾内容：

```json
{
  "scene_id": "jakarta_chinatown_05",
  "chapter": 1,
  "background_style": "linear-gradient(135deg, #1a1a3e 0%, #4a148c 100%)",
  "dialogues": [
    {
      "character": "Lina (印尼华人朋友)",
      "indo": "Kamu sudah belajar banyak hari ini! Besok, kita ke Bandung, ya?",
      "zh": "你今天学到很多啦！明天我们去万隆吧？",
      "vocab_hints": [
        {
          "word": "belajar",
          "meaning": "学习",
          "root": "ajar=教 → belajar=学习（前缀 ber- 表示进行某活动）",
          "culture_hook": "ber- 是印尼语最常见的前缀之一，第二章会大量遇到"
        }
      ]
    }
  ],
  "choices": [],
  "ending": {
    "title": "第一章完成！",
    "text": "你在雅加达华人区度过了充实的一天。\n从夜市的第一声 Selamat malam，到餐馆里自信地说出 Mau pesan Nasi Goreng——你已经迈出了第一步。\n\n但旅程才刚开始。Lina 提到的万隆，藏着什么故事？",
    "cta": "第二章：万隆茶园之旅 · 即将到来"
  }
}
```

**前端处理逻辑：** 当 `choices` 为空数组时，不渲染选项面板，改为渲染 `ending` 字段的结尾卡片（标题 + 正文 + CTA 文字）。

## 五、第一章内容规划

### 五个场景概览

| # | 场景 ID | 地点 | 核心学习内容 | 情绪节奏 |
|---|---------|------|-------------|---------|
| 1 | `jakarta_nightmarket_01` | 华人区夜市 · 街头摊位 · 夜晚 | 问候语：selamat malam/pagi/sore、apa kabar | 轻松好奇，初来乍到 |
| 2 | `jakarta_market_02` | 传统菜市场 · 蔬果摊位 · 白天 | 砍价：berapa、mahal/murah、蔬菜水果名 | 热闹嘈杂，有点紧张 |
| 3 | `jakarta_warung_03` | 华人餐馆 · 点餐 | 食物词汇密集：goreng 系列、makan、mau pesan | 放松享受，词汇爆发 |
| 4 | `jakarta_cny_04` | 华人社区活动中心 · 新年聚会 | 节日祝词、angpao、数字 1-10 | 温馨热闹，文化碰撞 |
| 5 | `jakarta_chinatown_05` | 华人区街头 · 夜晚 · 第一章结尾 | 复习本章高频词，引出第二章线索 | 意犹未尽，想继续 |

### 词汇复现目标

demo 阶段不需要精确追踪，但写脚本时心里要有数：

| 词汇 | 目标出现次数 | 覆盖场景 | 说明 |
|------|------------|---------|------|
| goreng | ≥ 6 次 | 场景 1、3、4、5 | 夜市小吃（pisang goreng）→ 餐厅菜单（nasi/mie goreng）→ 聚会食物 → 结尾复习 |
| makan | ≥ 5 次 | 场景 1、3、4 | 夜市问候（sudah makan?）→ 餐厅点餐 → 聚会用餐 |
| berapa | ≥ 4 次 | 场景 1、2、3 | 夜市问价 → 菜市场砍价 → 餐厅看菜单 |
| enak | ≥ 4 次 | 场景 1、3、4、5 | 夜市评价小吃 → 餐厅评价菜品 → 聚会聊天 → 结尾复习 |
| mau | ≥ 5 次 | 场景 1、2、3、4 | 高频意愿词，几乎每个场景的对话都自然用到 |

### 每个场景的对话轮数与选项

- 每个场景 **5-8 轮对话**，最后一轮之后展示选项
- 每个选项 **2-3 个**，选项文本用"印尼语 + 中文括号注释"格式
- 场景 5 的 `choices` 为空数组，渲染 `ending` 字段的结尾卡片（见第四节数据结构示例）

## 六、交互设计要点

### 对话推进

- 进入场景后显示第一轮对话（角色名 + 印尼语 + 中文）
- 用户点击"继续"按钮（或点击对话区域）推进到下一轮
- 当前轮的对话保留在屏幕上（聊天记录式），新对话追加在下方
- 最后一轮对话结束后，显示选项面板

### 生词高亮与弹窗

- 印尼语文本中，`vocab_hints` 对应的词以**下划线 + 强调色**显示
- 点击高亮词 → 弹出浮层，展示：词义、词根拆解、文化钩子
- 点击弹窗外区域或关闭按钮 → 收起弹窗
- 非高亮词点击无反应（demo 阶段不做全词库查询）

### 视觉占位方案

- **背景**：每个场景一个 CSS 渐变色，用颜色传递氛围
  - 夜市：深蓝 → 暖橙（`#1a1a3e → #e8a54b`）
  - 白天市场：浅黄 → 绿（`#F5E6CA → #A8D5A2`）
  - 餐厅：暖棕调（`#3E2723 → #8D6E63`）
  - 聚会：红金调（`#B71C1C → #FFD54F`）
  - 结尾：渐暗的蓝紫（`#1a1a3e → #4a148c`）
- **角色**：不显示头像，用角色名文字标签区分说话人
- **对话框**：深色半透明底 + 白色文字，参考视觉小说风格
- **整体风格**：干净、可读、不丑就行，不追求精美

## 七、开发计划：5 天搞定

| 天 | 任务 | 交付物 |
|----|------|-------|
| Day 1 | React 项目搭建 + 单场景渲染；**同步开始写场景 1-2 的 JSON 脚本** | 能看到一个场景的对话和中文对照 |
| Day 2 | 生词高亮 + 弹窗组件；**写场景 3 的 JSON 脚本**（词汇最密集，最花时间） | 点击高亮词能弹出释义 |
| Day 3 | 多轮对话推进 + 选项跳转 + 结尾场景渲染；**写场景 4-5 的 JSON 脚本** | 能从场景 1 一路点到场景 5 |
| Day 4 | 全部脚本填入项目跑通 + UI 微调 + 找印尼朋友校对内容 | 完整内容可体验 |
| Day 5 | 部署 Vercel + 找 3-5 人首轮测试 | 一个可分享的链接 |

**关于脚本工作量的说明：** 5 个场景 × 每场景 5-8 轮对话 = 25-40 轮对话内容，每轮包含印尼语原文、中文对照、生词标注。这是整个 demo 最耗时的部分，所以从 Day 1 起就同步写，不要留到最后一天。如果时间紧张，优先保证场景 1、3、5 的质量（开头、高潮、结尾），场景 2 和 4 可以适当缩减对话轮数。

## 八、测试反馈收集方法

### 怎么测

- 把 Vercel 链接发给测试用户，不做任何解释（观察他们能不能自己搞懂）
- 如果能当面测，坐在旁边观察，记录：哪里卡住了、哪里笑了、哪里跳过了
- 如果远程测，让他们录屏 + 语音，事后看

### 问什么（测完之后问，不要提前说）

1. 你觉得这个东西是干什么的？（检验定位是否清晰）
2. 有没有哪个地方让你想关掉页面？（找到流失点）
3. 你记住了哪几个印尼语词？（检验学习效果）
4. 如果有第二章，你会想继续吗？（检验留存意愿）
5. 你会把这个发给谁？（检验传播意愿）

### 不要问什么

- "你觉得这个产品怎么样？"（必然得到礼貌性肯定）
- "你会不会付费？"（demo 阶段问这个没意义）
- "UI 好不好看？"（demo 用的是占位设计，外观不是验证点）

## 九、这个 Demo 之后的决策树

```
Demo 做完，发给 10 个人测
        │
        ├─ 完成率 > 30%，有人主动问"第二章呢"
        │   → 信号积极，值得投入更多
        │   → 下一步：加后端、加登录、做第二章内容
        │
        ├─ 完成率 15-30%，反馈"还行但缺点什么"
        │   → 方向可能对，但体验有硬伤
        │   → 下一步：根据反馈迭代内容和交互，再测一轮
        │
        └─ 完成率 < 15%，没人主动传播
            → 这个形态不成立
            → 下一步：把精力收回来放到申研上，项目作为 portfolio 保留
```
